2. เจาะลึกจุดยุทธศาสตร์และ "ช่องว่าง" การทำ SaaS (In-Depth Blueprint)

การแยกส่วนตลาดเพื่อมองหาโอกาสในการสร้าง Product ตามหลักการลงทุน (INVEST) และแนวทางการทำรายได้ (Monetization)

The Revenue Engine

1 ฝั่งผู้ออกเอกสาร (Issuers)

กลุ่มเป้าหมายหลักในการสร้างกระแสเงินสด

🎯 Target Group (TH): มหาวิทยาลัย (ใบปริญญา), โรงพยาบาล (ใบรับรองแพทย์), โบรกเกอร์/TSD (แจ้งสิทธิผู้ถือหุ้น), และอสังหาริมทรัพย์ (ใบเสร็จรับเงิน/สัญญา)

⚠️ Deep-Dive Pain Point:

ระบบ Core System เป็นแบบ Silo ทีมผู้พัฒนาภายใน (In-house) ขาดความเชี่ยวชาญในการทำ Cryptographic Signing (W3C), จัดการ Key Management, และแปลง Data เป็นฟอร์แมต JSON-LD/VC

💡 SaaS Opportunity: "VC Issuance API Gateway"

แนวทางโปรดักส์: สร้าง Middleware API รับข้อมูลดิบ JSON จากลูกค้า ซอฟต์แวร์นำไป "ห่อ" เป็น VC, Sign ด้วย Private Key ผ่าน Cloud HSM, และอัปเดตลง VDR อัตโนมัติ (Plug-and-play)

💸 Monetization: Per-Transaction Tier (เช่น 0.50 - 2.00 บาท/ฉบับ) ยิ่งสเกลระดับ Mass อย่างใบแจ้งหนี้ รายได้ยิ่งเป็นทวีคูณ

Network Effect

2 ฝั่งผู้ตรวจสอบเอกสาร (Verifiers)

กลุ่มขยายฐานเพื่อดึงดูดผู้ใช้งานเข้าสู่ระบบ

🎯 Target Group (TH): ฝ่ายบุคคล (HR), สถาบันการเงินที่ต้องทำ Credit Scoring/KYC, แพลตฟอร์ม Onboarding

⚠️ Deep-Dive Pain Point:

VDR ในไทยกระจัดกระจาย (Fragmentation) ฝั่งคนตรวจไม่อยากเขียนโค้ดเพื่อไปต่อ API กับ VDR หลายๆ ที่เพื่อตรวจเช็กสถานะการ Revoke (เพิกถอน) ต้องการโซลูชันเดียวที่ฟันธงได้เลย

💡 SaaS Opportunity: "1-Click Verification Portal"

แนวทางโปรดักส์: สร้าง Registry Aggregator รวมทุก VDR ไว้ที่เดียว เมื่อ Verifier สแกน VC ระบบจะวิ่งไปตรวจ Cryptographic Proof, DID, และ Revocation List ทันที พร้อมออก Audit Report

💸 Monetization: Monthly/Annual SaaS Subscription (Seat-based) หรือ Pay-per-Verification API call

High Barrier to Entry

3 ผู้ให้บริการโครงสร้างข้อมูล (VDR)

กลุ่มผู้สร้างรากฐานระดับอุตสาหกรรม

🎯 Target Group (TH): สมาคมวิชาชีพ, Consortium ของกลุ่มธุรกิจ, หรืออุตสาหกรรมเฉพาะทางที่ต้องการสร้าง Trust Network

⚠️ Deep-Dive Pain Point:

การตั้ง Node VDR ให้มี Uptime 99.99% ตลอดเวลา และระบบการจัดการ Revocation List (สถานะเอกสารถูกยกเลิก) แบบ Low Latency มีต้นทุน Infrastructure และ Operation ที่สูงและซับซ้อนมาก

💡 SaaS Opportunity: "Managed VDR / Revocation Node"

แนวทางโปรดักส์: ระบบคล้าย AWS แต่เจาะจงเฉพาะ SSI โดยให้บริการเช่าพื้นที่ VDR Node ที่ปลอดภัยสูง จัดการระบบ Revocation ทันใจ (เช่น Cryptographic Accumulators)

💸 Monetization: Compute & Storage Resource Subscription + SLA Contracts ระดับองค์กร

B2B2C Interface

4 กระเป๋าเอกสารผู้ใช้งาน (Wallet Providers)

จุดเชื่อมต่อประสบการณ์ประชาชนทั่วไป

🎯 Target Group (TH): นิติบุคคลโครงการอสังหาฯ, องค์กรที่มีสมาชิกลูกค้า (Membership)

⚠️ Deep-Dive Pain Point:

การบังคับประชาชนโหลดแอปพลิเคชัน (App) ใหม่เพื่อเก็บ VC เป็นอุปสรรคสำคัญ (Friction) ในขณะที่ธุรกิจก็ไม่อยากลงทุนพัฒนาแอปพลิเคชัน Wallet ที่ปลอดภัยสูงขึ้นมาเอง

💡 SaaS Opportunity: "White-Label Wallet SDK / LINE Mini App"

แนวทางโปรดักส์: พัฒนา Wallet เป็น SDK ฝังเข้าแอปฯ เดิมของลูกค้า หรือใช้แพลตฟอร์มยอดนิยมอย่างสร้าง LINE Mini App ให้ผู้ใช้เปิดรับและเก็บ VC ได้ทันที

💸 Monetization: B2B Licensing & Setup Fee (องค์กรเป็นผู้จ่าย) + ค่า MAU (Monthly Active Users)

Market Strategy Conclusion

ในยุคที่ตลาดประเทศไทย (มิถุนายน 2026) พิสูจน์แล้วว่าหน่วยงานรัฐวิสาหกิจระดับชาติเริ่มใช้ VC Wallet ระดับชาติ การแข่งขันจึงไม่ใช่การสร้างแอปพลิเคชันกระเป๋าเงินเพื่อแย่งชิงประชาชน

จุดยุทธศาสตร์ (Moat) ที่สร้าง Margin สูงสุดคือฝั่ง B2B: การทำหน้าที่เป็น "ท่อประปา (Plumbing)" ให้องค์กรต่างๆ สามารถ ออกเอกสาร (Issuer API) จำนวนมหาศาล หรือ ตรวจสอบเอกสาร (Verifier Portal) ได้อย่างง่ายดาย โดยมีโมเดล Pay-Per-Transaction เป็นใบเบิกทางที่ลดแรงเสียดทาน (Friction) ในการตัดสินใจของฝ่ายจัดซื้อองค์กรได้ดีที่สุด

🏛️ VDR Competitor

แผนที่คู่แข่งเฉพาะฝั่ง Registry/VDR (Segment 3 ใน Market Overview) ทั่วโลก

ภาพรวม: ตลาด VDR แบ่งเป็น 3 กลุ่ม

ผู้เล่นที่แตะ "Registry" ในระบบนิเวศ VC ทั่วโลก แบ่งตามโมเดลธุรกิจได้ 3 กลุ่มใหญ่ ซึ่งมีความใกล้เคียงกับสิ่งที่ TKC ทำต่างกันไป

1️⃣ Pure-play Trust Registry/VDR

ทำ Registry เป็นสินค้าหลักโดยตรง — รายได้หลักมาจากการขาย/ให้เช่าบริการ Registry เอง ไม่ใช่ผลพลอยได้จากธุรกิจอื่น = คู่แข่งโดยตรงของ TKC VDR

2️⃣ Broader Identity Platform

Registry เป็นแค่โมดูลหนึ่งในแพลตฟอร์มใหญ่ที่ทำ Issuance + Verification + Wallet ด้วย ลูกค้าซื้อทั้งแพลตฟอร์ม ไม่ได้ซื้อ Registry แยก

3️⃣ Infrastructure/Hosting

🇪🇺 EUDI Wallet Research

European Digital Identity Wallet — กระเป๋าเอกสารดิจิทัลระดับสหภาพยุโรป บังคับใช้ตามกฎหมาย

คืออะไร

EUDI Wallet คือกระเป๋าเอกสารดิจิทัลที่ทุกประเทศสมาชิก EU ต้องจัดหาให้ประชาชนและธุรกิจใช้งานได้ ภายใต้กฎหมาย eIDAS 2.0 (European Digital Identity Framework) รองรับการออก/แสดง Electronic Attestation of Attributes (EAA) เช่น ตั๋ว บัตรสมาชิก ใบขับขี่ วุฒิการศึกษา — ถ้าออกโดย Qualified Trust Service Provider จะกลายเป็น Qualified EAA (QEAA) ที่มีผลทางกฎหมายเทียบเท่าเอกสารต้นฉบับ เทคโนโลยีเบื้องหลังใช้ W3C Verifiable Credentials, ISO/IEC 18013-5 (ขยายจาก mDL ไปใช้กับเอกสารประเภทอื่น) และ European Digital Identity Architecture Reference Framework (ARF)

หลักการสำคัญที่ระบุไว้ในเอกสารทางการของ European Commission คือ "everyone should always control their digital identity" — พลเมืองต้องพกพาตัวตนดิจิทัลข้ามพรมแดนในยุโรปได้อย่างไร้รอยต่อ โดยไม่เสียการควบคุมข้อมูลของตัวเอง เน้นความเป็นส่วนตัวและความปลอดภัยเป็นแกนหลักของโครงการ

Source: EU Digital Identity Wallet Home (European Commission), About the initiative (European Commission)

ที่มาและประวัติ (How it began)

2557 (2014): กฎหมาย eIDAS ฉบับแรก (Regulation (EU) No 910/2014) ให้ประเทศสมาชิกใช้ระบบยืนยันตัวตนอิเล็กทรอนิกส์แห่งชาติ (eIDs) เข้าถึงบริการรัฐข้ามพรมแดนได้ แต่เมื่อบริการออนไลน์ทั้งรัฐและเอกชนเติบโตขึ้นเรื่อยๆ ความกังวลเรื่อง privacy/surveillance ก็เพิ่มตาม ทำให้ต้องมีการอัปเดต

2567 (2024): วันที่ 29 ก.พ. 2567 European Parliament ลงมติผ่านกฎหมายด้วยคะแนน 335 เห็นด้วย, 190 ไม่เห็นด้วย, 31 งดออกเสียง (Regulation (EU) 2024/1183) มีผลบังคับใช้ 20 พ.ค. 2567

ตัวอย่างผู้ให้บริการฝั่งตรวจสอบ: Credence ID

Credence ID (Oakland, สหรัฐฯ) เป็นตัวอย่างบริษัทที่ทำฮาร์ดแวร์/ซอฟต์แวร์เฉพาะฝั่ง Verifier เพื่อเชื่อมช่องว่างระหว่างบัตรพลาสติกกับ mobile ID สินค้าหลักแบ่งเป็น 2 กลุ่ม

📱 Mobile ID

  • Tap2iD™ Verifier — อุปกรณ์ตรวจสอบเฉพาะทาง (standalone) รองรับทั้งบัตรจริงและ mobile ID
  • Tap2iD Mobile™ — ตรวจสอบ mobile ID ผ่านสมาร์ทโฟนโดยตรง ไม่ต้องมีฮาร์ดแวร์เพิ่ม
  • Tap2iD™ SDK — ผสาน mobile ID verification เข้าแอปพลิเคชันที่มีอยู่แล้ว
  • Tap2iD™ Cloud SDK — ตรวจสอบ mobile ID แบบ remote สำหรับเว็บ (นี่คือฝั่ง Online/Server Retrieval)
  • Verify with Credence™ Management Portal — endpoint management และ compliance reporting

👆 Biometric ID

  • CredenceECO™ — เครื่องอ่านไบโอเมตริกซ์พกพา กะทัดรัด ใช้งานนอกสถานที่
  • Credence-3™ — handheld แบบ multi-modal ทนทาน สำหรับปฏิบัติงานภาคสนาม/ลงทะเบียนระยะไกล
  • CredenceTAB4™ — แท็บเล็ตสำหรับลงทะเบียนและตรวจสอบไบโอเมตริกซ์
  • CredenceCONNECT™ — จัดการอุปกรณ์แบบรวมศูนย์ ตั้งค่า และดู analytics แบบ real-time

จุดที่สำคัญคือรองรับการตรวจสอบทั้ง Online และ Offline ตามที่มาตรฐาน ISO/IEC 18013-5 กำหนดไว้จริง:

🔌 Offline — Device Retrieval

แตะ NFC หรือสแกน QR code ระหว่างอุปกรณ์ผู้ถือบัตรกับเครื่องตรวจโดยตรง (peer-to-peer) ไม่ต้องต่ออินเทอร์เน็ต ไม่มีการส่งข้อมูลไปที่ศูนย์กลาง/หน่วยงานออกบัตรเลย เครื่องตรวจแค่มี public key ของหน่วยงานออกบัตรเก็บไว้ล่วงหน้าก็ตรวจลายเซ็นได้ทันที — เหมาะกับพื้นที่สัญญาณไม่ดี เช่น สนามกีฬา งานอีเวนต์

☁️ Online — Server Retrieval

ดึงข้อมูลยืนยันแบบ real-time ตรงจากหน่วยงานผู้ออกบัตร (Issuing Authority) เหมาะกับธุรกรรมที่ไม่มีคนคอยดูแล (unattended) หรือกรณีต้องการความมั่นใจสูงสุด — ผลตรวจของเครื่อง Tap2iD ยังส่งเข้า Verify with Credence™ Portal เก็บเป็นข้อมูล analytics แบบ real-time ให้ธุรกิจดูภาพรวมได้ด้วย

Source: Credence ID — Tap2iD, Credence ID Home

Offline vs Online ตามมาตรฐาน ISO/IEC 18013-5

มาตรฐาน ISO/IEC 18013-5 บังคับ ให้ mDL ทุกใบต้องรองรับ Device Retrieval (offline) เป็นพื้นฐาน ส่วน Server Retrieval (online) เป็นออปชันเสริม — คนละมาตรฐานกับ ISO/IEC 18013-7 (ประกาศ ก.ย. 2567) ที่ออกแบบมาสำหรับ remote verification โดยเฉพาะ (เช่น ยืนยันตัวตนผ่านเว็บโดยไม่ต้องเจอหน้ากัน)

Source: Dock — ISO 18013-5 Explained

ข้อจำกัดปัจจุบัน

mDL ยังไม่มี wallet เดียวที่ครอบคลุมทั้งประเทศ เพราะแต่ละรัฐทยอยเปิดเอง — ธุรกิจที่ต้องตรวจตัวตน (สนามกีฬา ร้านค้าปลีก หน่วยงานรัฐ ธนาคาร) จึงต้องรองรับหลาย wallet พร้อมกัน แนวทางที่แนะนำคือสร้างระบบตรวจสอบตาม standard ISO 18013-5 กลาง แทนที่จะไล่ตามอัปเดตของแต่ละ wallet ทีละเจ้า นอกจากนี้ตำรวจ หน่วยงานรัฐ และธุรกิจส่วนใหญ่ยังไม่ยอมรับ mDL แบบเต็มรูปแบบ — ปัจจุบันใช้งานได้แบบจำกัดขอบเขต

Microsoft Research — Entra Verified ID

สรุปแนวทางของ Microsoft ในการใช้ Verifiable Credential ภายในระบบนิเวศ Entra ID


1. Overview: Entra Verified ID

Entra Verified ID เป็นความสามารถด้าน การพิสูจน์ตัวตน (Identity Verification) ภายใน Microsoft Entra ID สร้างบนมาตรฐาน W3C Verifiable Credentials และ DID เดิมทีรองรับ did:ion (DID ที่ผูกกับ Bitcoin blockchain แบบ Layer-2) แต่ปี 2569 Microsoft ยกเลิกไปแล้ว เหลือ did:web (อิงกับ DNS/TLS ธรรมดา ไม่ใช้ blockchain) เป็น trust system เดียวที่ใช้งานได้จริง

⚠️ จุดสำคัญที่สุด: Microsoft ระบุไว้ชัดในเอกสารทางการว่า Verified ID "is an identity verification capability... not a traditional authentication method. It can't be used to satisfy authentication requirements like sign-in, MFA, or SSPR" — หน้าที่จริงคือ "พิสูจน์ว่าคุณเป็นใคร" ผ่านการเข้ารหัส ไม่ใช่ "อนุญาตให้ล็อกอิน" ส่วน sign-in ประจำวันจริงๆ Microsoft ผลักดันไปทาง Passkey/Passwordless แทน ซึ่งเป็นคนละกลไกกับ VC

Source: Microsoft Entra authentication overview


2. Use Cases ของ Microsoft

Use Case 1

🔑 Account Recovery

สำหรับตอนที่ผู้ใช้ทำ auth method หายหมด ล็อกอินไม่ได้เลย ระบบจะส่งไปหา IDV partner บุคคลที่สาม ให้สแกนบัตร/พาสปอร์ต + เช็คใบหน้าแบบ liveness (Face Check) จากนั้น IDV จะออก Verified ID ให้สดๆ ณ ตอนนั้น (ไม่ใช่ของที่ออกไว้ล่วงหน้าตั้งแต่สมัคร) เก็บใน Microsoft Authenticator แล้วส่งให้ Microsoft Entra ซึ่งทำหน้าที่เป็น Verifier ตรวจสอบ claim ให้ตรงกับบัญชี ถ้าผ่านจะได้ Temporary Access Pass ไปตั้งค่า auth method ใหม่

  1. กด "Recover your account"
  2. ส่งไปหา IDV partner (เช่น 1Kosmos, CLEAR)
  3. สแกนเอกสาร + Face Check
  4. IDV ออก Verified ID สดๆ
  5. Entra (Verifier) ตรวจสอบ → ออก Temporary Access Pass
  6. ผู้ใช้ re-enroll auth method ใหม่

Source: Account Recovery Overview — Microsoft Learn

🏢 Enterprise Onboarding

VC Framework — EBSI Hub

สรุปโครงสร้าง Verifiable Credentials Framework ของ EBSI (European Blockchain Services Infrastructure) — พาร์ทเนอร์ชิปของทุกประเทศสมาชิก EU, นอร์เวย์, ลิกเตนสไตน์ และ European Commission

EBSI VC Framework เป็นเอกสารและเครื่องมือกลางสำหรับใครก็ตามที่ต้องการเชื่อมระบบของตัวเองเข้ากับระบบนิเวศ Verifiable Credential ของสหภาพยุโรป แบ่งเนื้อหาออกเป็น 7 หมวดหลัก ดังนี้

Source: hub.ebsi.eu/vc-framework

1. EBSI W3C VCs and VPs

แนะนำ framework ของ Verifiable Credentials และ Verifiable Presentations ตามที่ W3C กำหนดไว้ เป็นจุดเริ่มต้นสำหรับทำความเข้าใจมาตรฐานที่ EBSI ใช้

Browse category →

2. Data Models

ข้อมูลเกี่ยวกับ data model ที่ใช้ใน EBSI รวมถึงโมเดลสำหรับสร้าง Trust Chain และโมเดลเฉพาะแต่ละ use case

Browse category →

3. DID Methods

DID Method ของ EBSI แยกเป็น 2 แบบ: DID Method for Legal Entities (นิติบุคคล) และ DID Method for Natural Persons (บุคคลธรรมดา) พร้อม security consideration เรื่อง private key

Browse category →

4. E-signing and e-sealing

แนวทางสำหรับ Issuer และ Holder ในการเซ็นและประทับตรา (seal) Verifiable Credential ให้ถูกต้องตามมาตรฐาน

Browse category →

5. Trust Model

อธิบาย Issuer Trust Model (เวอร์ชันปัจจุบัน v3 และเวอร์ชันถัดไป v4) วิธี onboard เข้า Trust Chain และ policy สำหรับการนำเสนอ/ขอ Verifiable Credential — เทียบได้กับแนวคิด Trust List ของ ETDA

Browse category →

6. Credential Status Framework

ภาพรวมวิธี revoke/เช็คสถานะ VC ของ EBSI พร้อมแนวทางเลือกกลยุทธ์ที่เหมาะสม, รูปแบบ Credential Status VC และ schema ที่รองรับ — เทียบได้กับ Bitstring Status List ที่ TKC VDR ใช้

Browse category →

7. Guidelines

คู่มือละเอียดสำหรับการทำ action สำคัญต่างๆ ภายในระบบนิเวศ EBSI เช่น การ verify credential, การ integrate wallet เข้ากับระบบ

Browse category →

ทำไมเรื่องนี้เกี่ยวกับ TKC VDR: โครงสร้าง 7 หมวดของ EBSI (DID Method, Trust Model, Credential Status Framework ฯลฯ) ใกล้เคียงกับ module ที่ TKC VDR ออกแบบไว้มาก (DID Registry, Trust Verification, Status/Revocation Service) — สามารถใช้เป็น reference architecture เทียบเคียงเวลาคุยกับ ETDA เรื่องมาตรฐานสากลได้

1. NIST SP 800-63-4 — Digital Identity Guidelines

มาตรฐานทางเทคนิค (ไม่บังคับตามกฎหมาย)

เอกสารแนวทางกลางที่ใกล้เคียง "มาตรฐานระดับชาติ" ที่สุดของสหรัฐฯ ออกโดย National Institute of Standards and Technology (NIST) กำหนด assurance level สำหรับ identity proofing, authentication, และ federation หน่วยงานรัฐบาลกลางต้องอ้างอิงมาตรฐานนี้เวลาจัดซื้อ/ออกแบบระบบยืนยันตัวตน แต่ไม่ใช่กฎหมายบังคับรัฐหรือเอกชน ฉบับล่าสุด (Revision 4) เพิ่มแนวทางป้องกันการฉ้อโกง, การรองรับ digital wallet ที่ผู้ใช้ควบคุมเอง (user-controlled), และ syncable authenticator (passkey)

เทียบกับ EU (โน้ตสั้นๆ): เทียบกับ EU: คล้าย ARF ของ EBSI ตรงที่เป็นมาตรฐานทางเทคนิคกลาง แต่ NIST ไม่มีสถานะบังคับทางกฎหมายเหมือน ARF ที่ผูกกับ eIDAS 2.0

Source: NIST Digital Identity Guidelines SP 800-63 Rev 4

2. NIST NCCoE SP 1800-42A

Practice Guide เฉพาะภาคการเงิน

ร่างคู่มือปฏิบัติ (draft practice guide) จาก National Cybersecurity Center of Excellence (NCCoE) ซึ่งเป็นหน่วยงานลูกของ NIST เผยแพร่ปี 2569 โดยร่วมมือกับพันธมิตรอุตสาหกรรมและรัฐบาล 29 ราย มุ่งแก้ปัญหาความปลอดภัย ความเป็นส่วนตัว และ interoperability สำหรับสถาบันการเงินที่ต้องการใช้ mDL ในกระบวนการยืนยันตัวตนลูกค้า (KYC) อ้างอิง NIST SP 800-63-4 เป็นมาตรฐานหลัก

เทียบกับ EU (โน้ตสั้นๆ): เทียบกับ EU: เจาะจงเฉพาะภาคการเงิน ต่างจาก EU ที่ EUDI Wallet ครอบคลุมทุกภาคส่วนพร้อมกันตั้งแต่ต้นด้วยกฎหมายเดียว

Source: NIST draft guidelines for financial firms — Biometric Update

3. DHS — Digital Immigration Credentialing

โครงการเฉพาะหน่วยงาน (Department of Homeland Security)

กระทรวงความมั่นคงแห่งมาตุภูมิ (DHS) เปิดโครงการพัฒนา "privacy-preserving digital credentialing ecosystem" ให้ครอบคลุมหน่วยงานลูกได้แก่ USCIS (สำนักงานตรวจคนเข้าเมืองและสัญชาติ), CBP (ศุลกากรและป้องกันชายแดน) และ DHS Privacy Office โฟกัสที่การออกเอกสารตรวจคนเข้าเมืองแบบดิจิทัลที่ตรวจสอบได้ (verifiable) โดยให้ผู้ถือควบคุมข้อมูลของตัวเอง

4. AAMVA — mDL Implementation Guidelines

5. SMART Health Cards / VCI (Vaccination Credential Initiative)

2. SGFinDex

Public Digital Infrastructure — ข้อมูลการเงิน

✅ มาตรการของประเทศ

เทียบกับ EU (โน้ตสั้นๆ): แนวคิดคล้าย Open Banking/PSD2 ของ EU แต่ผูกกับ Singpass โดยตรงเป็น NDI-native ไม่ใช่มาตรฐานเปิดแยกต่างหาก

4. OpenAttestation (OA)

Core Framework โอเพนซอร์ส (หัวใจของฝั่งเอกสาร)

🧰 เฟรมเวิร์ค/เครื่องมือ (สมัครใจ)

เฟรมเวิร์คโอเพนซอร์สที่ GovTech Government Digital Services พัฒนาเอง กลไกทำงานจริงมี 2 ชั้น: (1) คำนวณ Merkle root ของเอกสารเป็นชุด (batch) แล้วผูก root นี้ไว้กับเอกสารแต่ละใบ นำ root ไปบันทึกบน blockchain หรือให้ผู้ออกเอกสารเซ็นด้วย private key เพื่อพิสูจน์ที่มา (2) ยืนยันตัวตนผู้ออกเอกสารผ่าน DNS-TXT record — เช็คว่า domain ของผู้ออกมี TXT record ที่มีคำว่า "openatts" ระบุ address ของ documentStore ตรงกันหรือไม่ ถ้าตรง แปลว่าเอกสารนี้ออกจาก domain นั้นจริง วิธีนี้ทำให้ตรวจสอบเอกสารได้โดยไม่ต้องพึ่ง Certificate Authority กลาง v3.0 กำลังทำให้ compatible กับ W3C VC Data Model UNDP ยอมรับว่าเป็น Digital Public Good — เป็น "แกนกลาง" ที่ OpenCerts, HealthCerts, TradeTrust ทั้งหมดสร้างต่อยอดมา ตัวเฟรมเวิร์คเองเป็น open source ให้เลือกใช้ ไม่บังคับหน่วยงานใดต้องใช้

เทียบกับ EU (โน้ตสั้นๆ): เทียบเท่า ARF/EBSI Framework ของ EU แต่เป็นแกนเดียวที่แตกเป็นหลายผลิตภัณฑ์ ไม่ใช่ wallet รวมศูนย์แบบ EU

Source: OpenAttestation — Singapore Government Developer Portal, OpenAttestation GitHub — DNS-TXT ADR

5. OpenCerts

ผลิตภัณฑ์เฉพาะทาง — การศึกษา

🧰 เฟรมเวิร์ค/เครื่องมือ (สมัครใจ)

สร้างบน OpenAttestation ใช้ digitize ใบรับรองการศึกษา/ฝึกอบรมให้เป็นเอกสารตรวจสอบได้ ป้องกันการปลอมแปลง ผู้เรียนเก็บไฟล์ .oa ไว้เอง แชร์กับนายจ้าง/สถาบันอื่นได้โดยไม่ต้องขอสำเนารับรองใหม่ทุกครั้ง ตรวจสอบผ่านเว็บ opencerts.io ได้ฟรี ปัจจุบันมีสถาบันเข้าร่วมใช้งานจริงแล้ว 18 แห่ง (รวม NTU) — จำนวนที่จำกัดสะท้อนว่าเป็นการเข้าร่วมแบบสมัครใจ ไม่ใช่ทุกสถาบันถูกบังคับให้ใช้

เทียบกับ EU (โน้ตสั้นๆ): บทบาทเทียบเท่า EBSI multi-university pilot หรือ Digital Credentials Consortium ของสหรัฐฯ แต่ production มาตั้งแต่ ~2561

6. HealthCerts

ผลิตภัณฑ์เฉพาะทาง — สุขภาพ

ชุดมาตรฐานดิจิทัลและ schema สำหรับคลินิก/ห้องแล็บออกผลตรวจโควิด-19/ใบรับรองวัคซีนดิจิทัล เริ่มปี 2564 อ้างว่าเป็น health certificate บน blockchain รายแรกของโลก ขยายผลไปไกลกว่าที่คิด — ปัจจุบันมี สถานพยาบาลราว 420 แห่งใน 9 ประเทศ ใช้ HealthCerts และมี ผู้ให้บริการ (provider) 9 ราย ใน 5 ประเทศ ช่วยให้สถาบันการแพทย์ออกใบรับรอง (เช่น Kiyo, Knowledge Catalyst, 3DCerts) Singapore Airlines เป็นสายการบินแรกที่ยอมรับ HealthCerts เป็นส่วนหนึ่งของกระบวนการตรวจสอบสุขภาพผู้โดยสาร GovTech open-source โค้ดทั้งหมดเพื่อให้ประเทศอื่นเอาไปใช้ตามมาตรฐานเดียวกันได้

7. TradeTrust

ผลิตภัณฑ์เฉพาะทาง — การค้าระหว่างประเทศ

🧰 เฟรมเวิร์ค/เครื่องมือ (สมัครใจ)

ใช้ OpenAttestation เป็นฐาน "notarization" ให้เอกสารการค้า (เช่น bill of lading) ตรวจสอบที่มาได้อิสระโดยไม่ต้องพึ่งตัวกลาง ดูแลโดย IMDA ปัจจุบันเชื่อมโยงระบบนิเวศกว่า 50 แพลตฟอร์มการค้าดิจิทัล สถาบันการเงินรายใหญ่ และสายการเดินเรือระดับโลก มี pilot ใช้งานจริงแล้วหลายคู่ประเทศ: Singapore-India digital trade corridor (ส.ค. 2566), Singapore-Rotterdam (2 ท่าเรือ transhipment หลัก ทดลองส่ง eBL จริง) และที่สำคัญมี pilot พิสูจน์ว่า eBL จาก TradeTrust framework ใช้ได้แม้ในประเทศที่ยังไม่มีกฎหมาย MLETR รองรับ เช่นประเทศไทย — ประเทศที่มีกฎหมาย MLETR รองรับแล้วได้แก่ UK, ฝรั่งเศส, New York/Delaware (สหรัฐฯ), Abu Dhabi Global Market

เทียบกับ EU (โน้ตสั้นๆ): EU ยังไม่มีผลิตภัณฑ์ trade-document เฉพาะทางที่ mature หรือ cross-border จริงเท่านี้

Source: TradeTrust Factsheet — IMDA, Scaling digital trade: TradeTrust's journey since 2021 — WCO

9. e-Apostille Framework

กรอบมาตรฐานเอกสารทางการระหว่างประเทศ

🧰 เฟรมเวิร์ค/เครื่องมือ (สมัครใจ)

กรอบการรับรองเอกสารทางการที่ยึดมาตรฐาน W3C VC โดยตรง เพื่อให้เอกสารที่ออกในสิงคโปร์ (เช่น เอกสารบริษัท ใบรับรองต่างๆ) ได้รับการยอมรับ/ตรวจสอบได้ในระดับสากลตามอนุสัญญา Apostille (Hague Convention) โดยไม่ต้องผ่านกระบวนการรับรองเอกสารกระดาษแบบดั้งเดิมที่ใช้เวลานาน — ปัจจุบันใช้กับเอกสารบางประเภทที่เข้าเงื่อนไขเท่านั้น ไม่ใช่ทุกเอกสารราชการ

เทียบกับ EU (โน้ตสั้นๆ): แนวคิดคล้าย Qualified EAA (QEAA) ของ EU ที่ต้องมีผลทางกฎหมายเทียบเท่าเอกสารต้นฉบับ

Source: Digital identity — GovTech

🎯 มุมที่ TKC น่าจะเทียบเคียงได้ตรงที่สุด: สถาปัตยกรรมของหมวด C "core registry/engine เดียว (OpenAttestation) ให้ Issuer หลายประเภทต่อยอด" ใกล้เคียงกับแนวทางที่ TKC VDR วางไว้มากกว่าหมวด A/B ที่เป็นเรื่อง identity/authentication ล้วนๆ — และเช่นเดียวกับ OpenAttestation, TKC VDR เองก็ควรถูกมองเป็น "เฟรมเวิร์คที่ Issuer เลือกเข้าร่วม" มากกว่า "มาตรการบังคับ" ในช่วงเริ่มต้น ที่น่าสนใจคือ TradeTrust มี pilot ที่พิสูจน์แล้วว่าใช้กับประเทศไทยได้จริง (แม้ไทยยังไม่มีกฎหมาย MLETR รองรับ) — อาจเป็นช่องทาง cross-border ที่ TKC ศึกษาต่อได้

10. กรอบกฎหมาย (ไม่มีกฎหมายเฉพาะ)

อาศัยกฎหมายทั่วไปที่มีอยู่แล้ว

📜 บริบทกฎหมาย

จุดต่างจาก EU ชัดเจนที่สุด — สิงคโปร์ไม่มีกฎหมายเฉพาะสำหรับ Singpass/NDI เลย ดำเนินการภายใต้กฎหมายทั่วไป 4 ฉบับที่มีอยู่ก่อนแล้ว: Public Sector (Governance) Act (2561) กำกับการจัดการข้อมูลของหน่วยงานรัฐ, Personal Data Protection Act (2555) มาตรฐานคุ้มครองข้อมูลส่วนบุคคลภาคเอกชน, National Registration Act (2508) รากฐานระบบทะเบียนราษฎร์ที่ Singpass อิงอยู่, และ Electronic Transactions Act กำกับบริการรับรองอิเล็กทรอนิกส์ที่น่าเชื่อถือ (คล้ายบทบาทของ พ.ร.บ. ธุรกรรมทางอิเล็กทรอนิกส์ของไทย)

เทียบกับ EU (โน้ตสั้นๆ): EU ต้องออกกฎหมายใหม่ (eIDAS 2.0) พร้อม deadline บังคับ แต่สิงคโปร์ทำภายใต้อำนาจบริหารเดิม เคลื่อนไหวได้เร็วกว่ามาก
